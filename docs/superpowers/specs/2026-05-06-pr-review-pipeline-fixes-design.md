# PR review pipeline fixes — design

**Status:** Draft
**Date:** 2026-05-06
**Trigger case:** [Panenco/qiv#292](https://github.com/Panenco/qiv/pull/292) — the bot's own narrative said "Would APPROVE on this round" twice but the verdict header stayed at COMMENT, the PR ended with a stale "Claude Review — incomplete" banner, and a duplicate empty-body review was posted on the same SHA two seconds after the substantive one.

## Problem

The pipeline is producing review artefacts that contradict each other and decay PR cleanliness over multiple rounds. Eight concrete failure modes, observed on PR #292 and verified in code:

1. **Anti-upgrade verdict ladder.** Once round 1 lands at `COMMENT` for any reason — including a one-word doc nit — every subsequent round is pinned at `COMMENT` even when the bot's own per-PR verdict is `APPROVE` and every prior finding is `RESOLVED`. Source: `scripts/build-review.sh:690-698`.

2. **Empty review created as a side-effect of thread-reply.** The duplicate review on PR #292 was not a retry artefact — it was created by `scripts/post-review.sh:272-278`. The `addPullRequestReviewThreadReply` GraphQL mutation auto-creates a new review container when the user has no pending review, and the bot never has one. Each round-2 resolution loop iteration that posts an audit-trail reply ("✅ Resolved as of …") creates a fresh empty-bodied review. PR #292 had one resolved finding → one extra review. PRs with N resolved findings get N extra empty reviews. Compounding bug: `post-review.sh:190-195` *also* retries `POST /reviews` on non-zero exit without idempotency, which is a separate latent duplicate-post path.

3. **Stale crash banner.** `scripts/verdict-gate.sh:84-92` posts the "incomplete" notice via `gh pr comment` (a top-level issue comment). The dismiss-stale-reviews logic in `post-review.sh:18-30` only operates on the bot's REVIEW objects, never on its issue comments. Result: the red "incomplete" banner survives every successful follow-up run.

4. **Author dismissals are invisible to the ladder.** When the author dismisses a `REQUEST_CHANGES` review, the round-state artifact still records `verdict: REQUEST_CHANGES`. The next round runs the round-2 ladder against the dismissed verdict as if it were live.

5. **Self-contradicting review body.** The body narrative is generated by the LLM from its judgement of the per-PR diff; the verdict header is then post-processed by the ladder. When the ladder downgrades, body and header disagree. On PR #292 the body literally said "Would APPROVE on this round" while the header said COMMENT.

6. **Inline comments silently dropped when outside diff hunks.** `scripts/post-review.sh:122-145` parses only `+start,count` (RIGHT side) from hunk headers and filters comments not matching. `scripts/build-review.sh:1016` hardcodes `side: "RIGHT"` for every comment. Findings on deleted lines, near-but-not-in-hunk lines, or lines the LLM picked imprecisely vanish entirely. To the user this looks like "the comments went to the general section" — the review body posts but the inline annotations are missing.

7. **Reply / `resolveReviewThread` operations fail silently and brittlely.** `scripts/post-review.sh:198-368`:
   - GraphQL mutations end with `2>/dev/null || true` — failures cannot be diagnosed.
   - `reviewThreads(first:100)` has no pagination cursor; long-running PRs lose threads.
   - Match key is `path AND (line OR originalLine) AND author` — drifts after several rounds when GitHub sets `line: null` and `originalLine` no longer equals `prior_state.findings[].line_end`.

8. **Skill bloat.** Skill files total 109 KB / 1643 lines. The largest (`review-context-builder.md`, 25 KB / 387 lines) embeds imperative bash — diff splitter, file-skip regex, convention-file matcher — that the agent re-reads on every PR. This is token cost paid for deterministic logic that belongs in shell scripts.

## Non-goals

- Re-scoring PR #292's verdict retroactively — the fix changes future behaviour; the next push to that PR exercises the new ladder.
- Replacing the verdict-gate exit-code semantics (`APPROVE` exits 0, `REQUEST_CHANGES` exits 1) — those stay as-is.
- Reworking the round-1 vs round-2 fan structure or the dedup pass.
- Auth, secrets, or token rotation — out of scope.
- Retiring `bugbot.md` / `review-config.md` — they stay as the consumer-facing config surface.

## Design

Seven changes, shipping together as one rework. Numbering matches the failure-mode list above where it doesn't combine multiple modes; change 7 is a review-quality addition that doesn't map 1:1 to a failure mode but composes with the rest.

### 1. Verdict ladder — kill anti-upgrade, keep anti-downgrade

`scripts/build-review.sh:676-710`. Replace the `COMMENT)` branch:

```bash
COMMENT)
  if [ "$HAS_BLOCKING" = "true" ]; then
    VERDICT="REQUEST_CHANGES"
  fi
  # No anti-upgrade clause: if the per-PR verdict is APPROVE,
  # let it stand. The ladder's job is anti-downgrade only.
  ;;
```

Add a body-prepend step in the same file when the ladder overrides the per-PR verdict (any branch). Format:

```
> :information_source: **Verdict pinned to $FINAL_VERDICT** by the round-2 ladder
> (per-PR verdict was $PER_PR_VERDICT; $REASON). See [verdict ladder](.../README.md#round-1-vs-round-2).
```

Where `$REASON` is one of: `prior round had $N still-present blocking finding(s)`, `prior verdict was REQUEST_CHANGES with malformed resolution-status`, etc. Eliminates the "body says X, header says Y" contradiction.

### 2. Dismissal-aware prior state

Two surfaces:

**a. State artifact extension.** `scripts/build-review.sh` — when writing `/tmp/review-state.json`, also record:

```json
{
  "review_id": <REST id of the review we just posted>,
  "review_node_id": "<GraphQL node id, for thread matching>",
  ...
}
```

**b. Prior-state read step in workflow.** `.github/workflows/pr-review.yml`, prior_state step (~line 628). After loading `/tmp/prior-state/review-state.json`, fetch the current state of `review_id`:

```bash
PRIOR_REVIEW_STATE=$(gh api "repos/$REPO/pulls/$PR/reviews/$REVIEW_ID" --jq '.state' 2>/dev/null || echo "")
case "$PRIOR_REVIEW_STATE" in
  DISMISSED)
    echo "::notice::Prior review #$REVIEW_ID was dismissed by author — treating prior verdict as APPROVE for ladder purposes."
    jq '.verdict = "APPROVE" | .dismissed_by_author = true' /tmp/prior-state/review-state.json > /tmp/r.json && mv /tmp/r.json /tmp/prior-state/review-state.json
    ;;
esac
```

The dismissed-flag is also surfaced in the body: "Prior review dismissed by author; treating findings as accepted/false-positive."

Round-state schema additions:
- `review_id` (number, REST id) — added round 1+
- `review_node_id` (string, GraphQL id) — added round 1+
- `dismissed_by_author` (boolean, default false) — set during round-2 prior-state read
- `posted_comments` — see change 4 below

### 3. Stop creating empty review containers + idempotent POST + crash-banner-as-review

**a. Replace `addPullRequestReviewThreadReply` with REST `replies`.** `scripts/post-review.sh:272-278` and `:350-355` (the cross-bot variant). Switch from the GraphQL mutation that auto-creates a review container, to the REST `POST /repos/{owner}/{repo}/pulls/{pr}/comments/{comment_id}/replies` endpoint that already works for cross-bot replies at `:105`. The REST endpoint requires the parent comment's REST `databaseId` — fetch it in the same GraphQL `reviewThreads` query that finds the thread. Net effect: zero auto-created empty reviews per resolved finding.

```bash
# After matching $THREAD via reviewThreads(...){nodes{id, comments(first:1){nodes{databaseId}}}}
PARENT_CID=$(echo "$THREAD" | jq -r '.comments.nodes[0].databaseId')
gh api --method POST "repos/$REPO/pulls/$PR/comments/$PARENT_CID/replies" \
  -f body="✅ Resolved as of \`$HEAD_SHA\` (Claude review round-2 classifier confirmed)." \
  || echo "::warning::  $rid: reply post failed (continuing to resolve)"
# Then the existing resolveReviewThread mutation, unchanged.
```

When `posted_comments[].comment_id` is already available from prior state (change 4d), use it directly as `PARENT_CID` and skip the GraphQL `comments(first:1){nodes{databaseId}}` roundtrip.

**b. Idempotent POST.** `scripts/post-review.sh:180-195` — before POSTing the substantive review, check whether the bot already posted one on the current head SHA:

```bash
HEAD_SHA=$(git rev-parse HEAD)
EXISTING=$(gh api --paginate "repos/$REPO/pulls/$PR/reviews" \
  --jq "[.[] | select(.user.login == \"$BOT_USER\" and .commit_id == \"$HEAD_SHA\")] | length")
if [ "$EXISTING" -gt 0 ]; then
  echo "::notice::Bot already posted $EXISTING review(s) on $HEAD_SHA — skipping duplicate POST."
  exit 0
fi
```

Then drop the retry-on-failure loop. A single POST that fails is a real failure that the gate reports via `posting_error`; retrying without idempotency was the latent duplicate-post path.

**c. Crash banner as review.** `scripts/verdict-gate.sh:84-92` — replace `gh pr comment` with `gh api --method POST .../reviews` carrying:
- `event: "COMMENT"`
- `body` includes a stable HTML marker `<!-- claude-review-crash -->`

**d. Crash-review supersession.** `scripts/post-review.sh:18-30` — after posting the new substantive review successfully, find any prior review whose body contains `<!-- claude-review-crash -->` and edit its body via `PUT /reviews/{id}` to:

```
<!-- claude-review-crash superseded -->

_Superseded by a successful review run on `$HEAD_SHA`._
```

The crash review can't be deleted (no review-delete API), but editing its body to the superseded form removes the misleading red-banner content while keeping the audit trail. The next push to PR #292 retroactively cleans up its lingering banner.

### 4. No-comment-left-behind + LEFT-side support

**a. `scripts/build-review.sh:1001-1024`** — propagate a `side` field through findings. The reviewer skills already produce `path` + `line_start`/`line_end`; extend the JSON schema to optionally carry `side: "LEFT" | "RIGHT"`. When absent, default to `RIGHT` (current behaviour). Update `review-core.md` and `review-sweep.md` to set `side: "LEFT"` when the finding is on a deleted line in the diff. Build the comment payload with whichever side the finding declares.

**b. `scripts/post-review.sh:116-145`** — extend the hunk-line builder to record LEFT-side lines too. Hunk headers carry both `-start,count` (LEFT) and `+start,count` (RIGHT); parse both, key the validation map by `path:line:side`.

**c. `scripts/post-review.sh:140-145`** — when validation drops a comment, do not silently delete. Append it to `/tmp/review-body.md` under a new section:

```markdown
### Findings outside diff hunks ($N)

These findings reference lines outside the PR's diff hunks (often deleted lines or context just outside the change window). Inline comments cannot be posted on those lines.

- **[BUG] `path/to/file.ts:42`** — Title of finding
  Reasoning paragraph.
```

Storage of dropped comments goes to `/tmp/dropped-comments.json` so the round-state artifact can include them in `posted_comments` (with `posted_inline: false`) for round-2 resolution checking.

**d. `posted_comments` round-state field.** Schema:

```json
"posted_comments": [
  {
    "finding_id": "c1",
    "comment_id": 1234567,        // null when posted_inline=false
    "node_id": "PRRC_xxx",         // null when posted_inline=false
    "path": "src/foo.ts",
    "line": 42,
    "side": "RIGHT",
    "posted_inline": true
  },
  ...
]
```

Findings dropped to the body (`posted_inline: false`) carry null `comment_id`/`node_id`. Round-2 thread resolution (change 5) only attempts to resolve threads when `posted_inline: true`; body-only findings have no thread to resolve and are surfaced via the resolution-checker's classification in the body's "Since previous review" section. This is the foundation for change 5.

### 5. Fingerprint-based thread resolution + visible failures

**a. `scripts/post-review.sh:198-292`** (own-thread resolve, round 2). Replace the path+line match with a `databaseId`-based primary match:

```bash
# Primary: match by stored comment_id
TARGET_ID=$(jq -r --arg id "$rid" '.posted_comments[]? | select(.finding_id == $id) | .comment_id' /tmp/prior-state/review-state.json)
THREAD=$(echo "$THREADS_JSON" | jq -c --argjson cid "$TARGET_ID" \
  '.[] | select(.comments.nodes[0].databaseId == $cid and .isResolved == false)' | head -n1)

# Fallback: path+line match (existing logic) for old prior states without posted_comments
if [ -z "$THREAD" ]; then ... existing path-line match ... fi
```

**b. Pagination.** Replace single `reviewThreads(first:100)` with cursor-based pagination, capped at e.g. 500 threads with a `::warning::` if exceeded.

**c. Visible failures.** Replace every `2>/dev/null || true` on GraphQL mutations with:

```bash
GQL_RESULT=$(gh api graphql -f query='...' -f threadId="$TID" 2>&1) || {
  echo "::warning::resolveReviewThread failed for thread $TID: $GQL_RESULT"
}
```

The `2>&1` capture means a mutation failure logs the error body (auth scope problems, "thread is archived", etc.) instead of vanishing.

Same three changes apply to the cross-bot reply at `post-review.sh:97-108` and the other-bot resolution loop at `:300-368`.

### 6. Skill trim with recall-regression gate

Move imperative bash out of skill prose into `scripts/`. The skill becomes a short prompt that calls the script.

| Skill | Current | Target | What moves |
|---|---|---|---|
| `review-context-builder.md` | 25 KB / 387 lines | ~12 KB | Diff splitter (Python block, `SKIP_PATTERNS`, per-file chunking) → `scripts/split-diff.sh`. Convention-file matcher → `scripts/match-conventions.sh`. Round-2 since-last diff/chunks → `scripts/since-last.sh`. |
| `review-test-planner.md` | 15 KB / 263 lines | ~9 KB | Technical-change detector (PR title/body keyword scan, dependency-manifest delta, move/rename ratio) → `scripts/detect-technical-change.sh`. Skill keeps the *judgement* prose: scenario shape, scope rules, when to skip. |
| `review-functional-tester.md` | 14 KB / 235 lines | ~10 KB | Pull common Playwright invocation patterns into a single "auth + nav primer" included by reference; skill keeps the "what to verify" prose. |
| `review-core.md` + `review-sweep.md` | 16 + 9 KB | 12 + 8 KB | Deduplicate methodology framing they share. Each keeps its own focus prose (correctness/security vs. consistency/perf) but drops shared boilerplate. |

**Hard rule, per project preferences:** no model demotions, no max-turns reductions. The trim must not change which model runs which role and must not lower any `--max-turns` ceiling. Recall is the north star.

Total ≈ 40% reduction (109 KB → ~65 KB) without touching review judgement quality.

**Recall-regression gate.** Before merging the trim, build a `tests/fixtures/historical-prs/` corpus of 8-10 historical real PRs spanning: a refactor, a feature, a bug-fix, a docs change, a CI/workflow change, a test-only change, a dependency bump, a security-relevant change. For each, capture the union of *true-positive* findings the previous skill versions produced (re-classified by hand: which findings were genuinely useful vs. noise).

Run the trimmed skills against the same corpus. The trim is **only accepted if recall on the true-positive set is ≥95%**. Any historical true-positive finding the trimmed skill misses must be diagnosed: either (a) the dropped prose was load-bearing — re-inline it, or (b) the prose was redundant with a different surviving section — accept the recall stays at 100%. Drops below 95% or unexplained absences block the trim PR.

A complementary precision check: count notes/minor findings on the same corpus. The trim should not *raise* false-positive volume either; if precision drops, that's also a regression.

Each extracted script is unit-testable in `tests/scripts/`.

### 7. Reviewer-prose hygiene: severity calibration + line accuracy

Two prose additions to `skills/review-core.md` and `skills/review-sweep.md`. These belong to the rework because they directly raise review *output* quality the user is asking for, and they compose with the no-comment-left-behind change (4) and the verdict ladder (1) — together they ensure the bot stops blocking PRs on calibration mistakes.

**Severity calibration rules.** Add an explicit rubric at the top of each skill's "What to report" section. Concrete cutoffs:

- **`critical`** — would corrupt data, allow unauthorised access, or break production deploys. The reviewer must demonstrate the failure mode, not just suspect it.
- **`major`** — runtime crash, observable wrong behaviour for a user-reachable code path, missed acceptance criterion, security regression that needs a CVE-class trigger to exploit.
- **`minor`** — code-quality issues that don't change behaviour: dead code, missed convention, weak typing, missing test for a non-trivial branch.
- **`note`** — doc/comment/identifier nits, readability suggestions, "you might consider …" style observations. Notes do not block APPROVE under the new ladder; they post inline so the developer sees them but never gate the verdict.

The PR #292 round-1 finding (`mapfile -t issues < <(gh pr view …)` allegedly swallowing failures) was over-graded as `critical`/blocker. Process substitution does lose the `gh` exit code, but the surrounding script handled "no issues" cleanly — the actual failure mode required AC-violating misconfiguration that the reviewer didn't demonstrate. With the rubric above this is at most `minor` (defensive scripting suggestion), or a `note`. The PR #292 round-2 finding (`@qiv/api-client` vs `@qiv/api-sdk` in a doc paragraph) is a textbook `note`. Neither would block APPROVE under the new ladder.

**Line-accuracy rules.** Add a "Pointing at the right line" section. The reviewer must:

1. Cite a line that exists in the diff hunks (RIGHT-side for added/modified lines, LEFT-side for removed lines). The bot now supports both via change 4.
2. When the bug is on a line that exists in the file but is not in the diff, point at the *closest line that IS in the diff* and explain in `reasoning` that the bug is at line N in the unchanged region. Do not invent line numbers.
3. When the finding is structural ("the new module lacks tests"), point at the topmost added line of the module — never line 1 of an unrelated file.
4. Multi-line ranges (`line_start` / `line_end`) are preferred when the finding spans logic. Cap at 10 lines (existing rule, retained).

These rules don't *replace* the change-4 fallback (findings still land in the body if the line is genuinely outside the diff), they *reduce how often the fallback is needed* — which keeps inline annotations dense, where they belong.

## Migration

Single rollout. Bumps the pipeline minor version (`v2.0.x` → `v2.1.0`), no breaking change to caller-workflow shape.

Round-state schema additions (`review_id`, `review_node_id`, `dismissed_by_author`, `posted_comments`) are forward-compatible: round-2 reads with `// empty` / `// false` defaults, so a v2.1 run reading a v2.0 prior state degrades to the existing path-line matcher and treats `dismissed_by_author` as false. Once both runs are v2.1, the new fingerprint matcher kicks in.

Verdict-gate change is fully internal.

Consumer-facing pieces (parallel updates required, per saved feedback):
- `README.md` — verdict ladder section ("Round 1 vs round 2") needs a one-paragraph update; new severity-rubric note explaining `note` never blocks; brief mention that `<!-- claude-review-crash -->` markers identify auto-superseded crash reviews.
- `prompts/setup-review.md` — same updates.
- Migration block can stay; the v1→v2 section doesn't need re-edits.

## Testing

Unit-testable pieces, all under `tests/`:

- `tests/verdict-ladder.bats` — synthesise prior-state fixtures, assert the new ladder produces APPROVE when prior=COMMENT + per-PR=APPROVE + all-resolved. Cover anti-downgrade cases (prior=REQUEST_CHANGES + still-present blockers stays REQUEST_CHANGES). Cover the body-prepend-on-override rationale.
- `tests/dismissal-awareness.bats` — fixture with a dismissed prior review; assert ladder treats prior as APPROVE and emits the `::notice::`.
- `tests/post-review-idempotent.bats` — mock `gh api .../reviews` to return existing review on head SHA; assert no second POST happens. Separate case: `addPullRequestReviewThreadReply` is never called (the REST `replies` endpoint is used instead).
- `tests/diff-hunk-validation.bats` — fixture with comments inside hunk (RIGHT), inside hunk (LEFT, deleted line), outside hunk (near-miss). Assert the first two are kept inline, the third lands in the body's "Findings outside diff hunks" section.
- `tests/crash-banner-supersession.bats` — fixture with a prior crash review carrying the `<!-- claude-review-crash -->` marker; assert post-review.sh edits its body to the superseded form after a successful POST.
- `tests/scripts/split-diff.bats` — direct test of the extracted diff splitter.
- `tests/scripts/detect-technical-change.bats` — direct test of the extracted detector.
- `tests/scripts/match-conventions.bats` — direct test of the extracted matcher.
- `tests/scripts/since-last.bats` — direct test of the round-2 since-last computation.

**Recall-regression corpus** (gates change 6): `tests/fixtures/historical-prs/` with 8-10 PR snapshots and a hand-curated true-positive set per PR. CI step that re-runs the trimmed skills against the corpus and asserts ≥95% recall on the true-positive set, with no precision regression.

Manual validation: run the local dogfood test against a small fixture PR (the `pipeline_ref` + caller-workflow setup already supports this) and verify:
1. A trivial round-2 fixing a prior nit → APPROVE.
2. A PR with a finding on a deleted line → comment posts inline on the LEFT side.
3. A PR producing a finding outside any diff hunk → "Findings outside diff hunks" section appears in the body, no comment is silently dropped.
4. A simulated crash followed by a successful run → no lingering crash banner (the crash review's body is superseded).
5. A round-2 review that resolves a prior finding → exactly one new review is created (the substantive one); no empty-body container appears.
6. An author-dismissed prior REQUEST_CHANGES → the next round can land on APPROVE if the per-PR verdict is APPROVE.

## Risk + open questions

- **Anti-upgrade removal could let a real round-1 blocker promote to APPROVE if the LLM forgets to repeat it in round 2.** Mitigation: the round-2 resolution-checker classifies every prior finding explicitly; STILL_PRESENT blockers escalate via the existing `HAS_BLOCKING=true → REQUEST_CHANGES` rule. The anti-upgrade clause was belt-and-braces against the resolution checker missing things; we keep the braces (anti-downgrade) and let the per-PR ladder do its job. The `tests/verdict-ladder.bats` matrix exercises this: prior=REQUEST_CHANGES + STILL_PRESENT blocker + per-PR=APPROVE must stay REQUEST_CHANGES.
- **Idempotency check is best-effort** — if the GET fails, the POST proceeds. Acceptable: at worst we revert to today's behaviour on that one push. No regression.
- **Skill trim could reduce recall** if extracted scripts behave subtly different from the prose, or the prose dropped a load-bearing rule. Mitigation: the recall-regression gate (≥95% on the historical corpus, no precision regression). Drops below 95% block the trim PR. The trim is structurally low-risk because we only extract *imperative bash* into scripts; the *judgement prose* the LLM uses to decide what to flag stays unchanged.
- **Severity rubric could be misapplied** — LLMs sometimes drop everything to `note` to look agreeable. Mitigation: the recall-regression corpus also tracks severity distribution; trims that systematically downgrade real bugs to `note` regress the corpus's blocker count and are rejected.
- **REST `replies` endpoint requires the parent comment's REST `databaseId`.** When `posted_comments[].comment_id` is unavailable (round 1 prior state from a v2.0 run), we fall back to the GraphQL thread query that returns `comments(first:1){nodes{databaseId}}`. Both paths are covered.
- **Crash review can't be deleted** — only edited. The superseded body still occupies a slot in the PR's review list, just with a non-misleading body. Acceptable: a one-line "_Superseded by …_" review is far better than today's red banner.
- **Open:** should `dismissed_by_author = true` carry over indefinitely, or only one round? Default: one round (the act of dismissing acknowledges *that round's* findings; subsequent rounds get fresh treatment). Revisit if real cases show this is wrong.
- **Open:** for the recall corpus, should we use historical PRs from the qiv repo specifically, or a mixed set across consumer repos? Default: qiv-only for the first iteration (largest signal source), expand later. Revisit if qiv-only over-fits the trim to one stack.
